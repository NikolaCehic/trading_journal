# UX Audit — 2026-04-25

**Auditor:** Claude Opus 4.7 (1M context), driving Chromium via Playwright against `pnpm dev` at `localhost:3000` with the demo user (12 positions, 1 finding seeded).
**Method:** Headless walkthrough of every authenticated route and flow, full-page screenshots at each meaningful state, structural inspection of DOM (column headers, ARIA labels, empty-state copy, CTAs, focus order), zero console errors and zero non-2xx HTTP across the run.
**Output:** `scripts/ux-out/` — 15 screenshots + this doc. Reproduce with `pnpm tsx scripts/ux-audit.ts`.

---

## Methodology

The audit looked for friction at four layers:

1. **Information scent** — does the user know what to do next at every state?
2. **Discoverability** — are the app's value-add features (detectors, plans, digest) findable from where the user actually looks?
3. **Feedback** — does the user know an action succeeded, failed, or is in progress?
4. **Cross-flow handoff** — does completing one flow guide the user into the next?

The two specific issues you raised — (a) no post-import guidance, (b) no detector visibility on `/trades` — are both real. They're not isolated; they're symptoms of a pattern: **the app's core value loop (import → analyze → spot patterns → adopt rules → review weekly) is implemented but not narrated**. Each step works in isolation; nothing pulls the user through the loop.

## Findings — by severity

### HIGH (6) — fix before anything else

#### H-01 · No per-row finding indicator on `/trades`

**Files:** `app/routes/(app)/_layout/trades/index.tsx`, query path in `src/server/trades.ts`

**Observation.** The trades table has 11 columns: Symbol, Side, Opened, Closed, Hold, Size, Entry, Exit, P&L $, P&L %, Tags. **There is no column or chip showing whether a position triggered any detector finding.** The seeded demo user has 12 positions and 1 active "Revenge trading" finding referencing two of those positions — but on `/trades`, those flagged rows look identical to the other 10. The only way to discover which trades have findings is to open each one and click the (unlabeled) "Findings" tab.

**Why it matters.** This is the app's headline differentiator — "Every pattern is deterministic and evidence-linked" (your public landing copy). If users can't see which trades the patterns flagged without clicking, the differentiator is invisible.

**Visual evidence.** `scripts/ux-out/02a-trades.png` — column headers row clearly shows no Findings/Flags column. `scripts/smoke-out/01-dashboard.png` — Findings sidebar shows "Revenge trading · 1 instance(s) of opening a new position within 15 minutes of a losing close" but you can't get from that finding to its underlying trades in one click from the trades list.

#### H-02 · No "flagged trades" filter on `/trades`

**Files:** same as H-01.

**Observation.** Filter row offers: All / Spot / Perp · Any / Long / Short · All P&L / Winners / Losers. Cannot filter to "only positions with findings" or "with revenge-trading flag." For users who imported 500 trades and the dashboard says "23 flagged" — there's no path to those 23.

#### H-03 · No post-import handoff — user is stranded on `/import`

**Files:** `app/routes/(app)/_layout/import.tsx`, both `CsvUploadCard` and `HLWalletCard`.

**Observation.** When a CSV import completes successfully, the import history table at the bottom of the page updates (one new row, status `complete`). That's the only feedback. No success toast, no "View your N trades →" CTA, no automatic redirect, no inline "Now go to /trades to see them" message. The import row is a passive log entry — clicking it does nothing.

**Visual evidence.** `scripts/ux-out/05a-import.png` — Import history shows the seeded binance-csv import (24 rows, status `complete`) as a non-interactive row.

**Code reference.** The import handlers (`startCsvImport` line 90, `startWalletImport` line 140) return `{ importId }` and the UI shows a toast with the import ID — useful for engineers, useless for users.

#### H-04 · Import-history rows aren't clickable / don't link to imported trades

**Files:** `app/routes/(app)/_layout/import.tsx`, history table render section.

**Observation.** The history table has columns: Date, Source, Status, Rows, Skipped. None are interactive. No "view trades" link, no per-import deep-link to `/trades?importId=...`, no per-account filter ("show only trades from this Hyperliquid wallet"). Once an import lands in history, it's a dead row.

#### H-05 · "Findings" tab on trade detail has no count badge

**Files:** `app/routes/(app)/_layout/trades/$positionId.tsx`.

**Observation.** Trade detail has a tab bar: Notes | Tags | Findings | Coach. Each tab has only the label — no `(0)` or `(1)` counter. A user opening a trade has no idea whether the Findings tab is empty until they click it. Same for Notes (could show "·" when present), Tags, Coach.

**Visual evidence.** `scripts/ux-out/EXTRA-trade-detail.png` — tab labels rendered as `Notes Tags Findings Coach` with no badges.

#### H-06 · Dashboard lacks an `<h1>` heading

**Files:** `app/routes/(app)/_layout/dashboard.tsx`.

**Observation.** The page jumps straight from the demo banner / TopBar into the date-range chip row + KPI tiles. There is no `<h1>Dashboard</h1>` (or "Last 30d overview", or whatever the page's purpose is). All the other sections (Equity curve, Asset breakdown, Findings, Time-of-day P&L) have `<h2>`-style titles via card headers — but the page itself has no top-level heading.

**Why it matters.** Two reasons: (a) screen readers have no anchor for the page; (b) visual hierarchy on first load points at "30d / 7d / 90d" as the first text, which is a filter, not an orientation. New users see KPI numbers without a label that tells them what they're looking at.

### MEDIUM (3) — fix in the next pass

#### M-01 · Predicate editor on `/detectors/new` has no templates or examples

**Files:** `app/routes/(app)/_layout/detectors/new.tsx`.

**Observation.** A user clicks "+ New detector" and is dropped into a from-scratch predicate editor: Name, Title, Severity (Info/Warning/Critical), Conditions (Match ALL/ANY/NOT), one row of `PnL (USD) < 0` placeholder. Below that: "Enter at least one condition value to see a live preview." No starter templates ("Loss > 1R", "Held overnight"), no "duplicate a built-in detector to learn the syntax", no link to docs.

**Why it matters.** Custom detectors are a power-user feature. Power users will figure it out, but the friction blocks the median user from ever creating their first one. The 12 built-in detectors are already serializable JSON predicates — surfacing them as starting points is mostly a copy + button.

**Visual evidence.** `scripts/ux-out/04b-detectors-new.png`.

#### M-02 · `/digest` has no orientation copy for first-time visitors

**Files:** `app/routes/(app)/_layout/digest/index.tsx`.

**Observation.** Page title: "Digest". Subtitle: "Preview this week's digest — exactly what ships Sunday at 22:00." Then a loading skeleton + "Composing…". For a user who's never seen one, that's not enough: what IS a digest? When does it actually ship? How does it relate to detectors and findings? Why "22:00" — local time? UTC? Their timezone?

**Visual evidence.** `scripts/ux-out/06-digest.png`.

#### M-03 · `/settings` has no section headings

**Files:** `app/routes/(app)/_layout/settings/index.tsx`.

**Observation.** The DOM scan returned `h2/h3 count = 0`. Settings is a list of toggle rows (Timezone, Digest enabled, Built-in detector toggles) without visible groupings. For 3-5 settings it's manageable; as soon as more land it'll be a wall.

## Cross-cutting patterns

### Pattern A: The value loop is unnarrated

The intended flow is:

```
import → trades (with findings highlighted) → trade detail (read finding, write note,
adopt rule) → digest (read AI summary of week's findings + adopted rules)
```

Each route works. Nothing tells the user the route is part of a flow. There are no transitional CTAs ("Next: see your trades → ", "1 detector flagged 2 of these trades — Review →", "Adopt this rule for next week's digest"). Every navigation hop is initiated by the user clicking a top-nav link they have to think about.

**Fix shape.** Two surfaces matter most:
1. **`/import` after success** → success toast/banner with `View N new trades →` linking to `/trades?source=<importId>`.
2. **`/trades` row** → small finding badge on rows that have findings, plus a filter pill `Flagged (3)` next to All P&L / Winners / Losers.

These two changes alone close 80% of the "what now?" gap.

### Pattern B: Findings live in two universes

`Finding` rows exist in the DB and reference position IDs via `referencedPositionIds: text[]`. The UI surfaces them in three places:
1. Dashboard sidebar — top 5 findings, no link from finding → trade
2. Trade detail "Findings" tab — per-trade, but no count badge in the tab strip
3. NOT on the trades list

The data model is correct. The UI just doesn't join the two sides. Adding a single query `SELECT positionId, count(*) FROM finding WHERE userId = ? AND derivationVersion = ? AND positionId = ANY(referencedPositionIds)` (or a derived column on `position`) and rendering a chip on the `/trades` row makes the connection visible everywhere.

### Pattern C: The first 60 seconds aren't designed

A new user lands on `/dashboard` after sign-in. With no data: empty KPI tiles, no equity curve, no findings. With seeded data: numbers that aren't labeled. There is no "Welcome — start by importing your first trades →" empty state, no progressive onboarding, no "Day 1" tutorial banner. The public landing page is gorgeous (`scripts/ux-out/08-landing-public.png`); the authenticated landing is anonymous.

**Fix shape.** Replace the dashboard empty state with a 3-step setup checklist:
- ✅ Sign in
- ⬜ Import your first trades — `[Import →]`
- ⬜ Review your first finding — appears after derivation finishes
- ⬜ Adopt your first rule — appears after first finding shown

The checklist hides itself once all are done. ~1 day of work, big perceived-quality lift.

## What this audit deliberately does NOT cover

- **Mobile responsive** — only checked at 1440×900 desktop viewport.
- **Performance** — TTFB, FCP, JS bundle size: separate audit.
- **A11y beyond what shipped in Wave 1** — focus rings, contrast tokens, htmlFor pairs are confirmed from the swarm plan; full WCAG sweep not redone.
- **Per-detector content** — the 12 built-in detectors have descriptions but no detector-detail page that explains the math/threshold; out of scope.
- **Sign-in flow with real Google OAuth** — used demo session for automation. Real sign-in tested manually earlier in this conversation.
